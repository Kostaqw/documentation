cmake

Синопсис

Создание сборочной системы проекта

cmake [<options>] -B <path-to-build> [-S <path-to-source>]
cmake [<options>] <path-to-source | path-to-existing-build>

            
#Сборка проекта
cmake --build <dir> [<options>] [-- <build-tool-options>]
            
#Установка проекта
cmake --install <dir> [<options>]
            
#Открытие проектаcmake
 --open <dir>
            
#Запуск скрипта
cmake [-D <var>=<value>]... -P <cmake-script-file>
            
#Запуск командной утилиты
cmake -E <command> [<options>]
            
#Запуск инструмента поиска пакетов
cmake --find-package [<options>]
            
#Запуск предустановленного рабочего процесса
cmake --workflow [<options>]
            
#Просмотр справки
cmake --help[-<topic>]

Описание

Исполняемый файл cmake представляет собой интерфейс командной строки для кросс-платформенной системы генерации сборки CMake. В разделе "Синопсис" приведены различные действия, которые инструмент может выполнять, как описано в следующих разделах.

Чтобы собрать программный проект с помощью CMake, сначала создайте сборочную систему проекта. Опционально, вы можете использовать cmake для сборки проекта, установки проекта или просто запустить соответствующий инструмент сборки (например, make) напрямую. cmake также может использоваться для просмотра справки.

Другие действия предназначены для использования разработчиками программного обеспечения, которые пишут скрипты на языке CMake для поддержки своих сборок.

Для графических интерфейсов, которые могут быть использованы вместо cmake, см. ccmake и cmake-gui. Для интерфейсов командной строки для тестирования и упаковки в CMake, см. ctest и cpack.

Для получения дополнительной информации о CMake, ознакомьтесь с ссылками в конце этого руководства.

Введение в сборочные системы CMake

Сборочная система описывает, как собрать исполняемые файлы и библиотеки проекта из его исходного кода с использованием инструмента сборки для автоматизации процесса. Например, сборочная система может быть Makefile для использования с командной строкой или файлом проекта для интегрированной среды разработки (IDE). Чтобы избежать поддержки множества таких сборочных систем, проект может абстрактно определить свою сборочную систему с использованием файлов, написанных на языке CMake. Из этих файлов CMake генерирует предпочтительную сборочную систему локально для каждого пользователя через бэкэнд, называемый генератором.

Для генерации сборочной системы с помощью CMake необходимо выбрать:

Дерево исходного кода (Source Tree)

Верхний уровень каталога, содержащий исходные файлы, предоставленные проектом. Проект определяет свою сборочную систему с использованием файлов, как описано в руководстве cmake-language(7), начиная с файла верхнего уровня, названного CMakeLists.txt. Эти файлы определяют цели сборки и их зависимости, как описано в руководстве cmake-buildsystem(7).

Дерево сборки (Build Tree)

Верхний уровень каталога, в котором будут храниться файлы сборочной системы и артефакты сборки (например, исполняемые файлы и библиотеки). CMake создаст файл CMakeCache.txt для идентификации каталога как дерева сборки и хранения постоянной информации, такой как параметры конфигурации сборочной системы.

Чтобы поддерживать чистоту дерева исходного кода, выполните сборку вне исходных файлов, используя отдельное дерево сборки. Также поддерживается сборка в исходном каталоге, при которой дерево сборки размещается в том же каталоге, что и дерево исходного кода, но это не рекомендуется.

Генератор (Generator)

Этот выбор определяет тип сборочной системы, которую необходимо создать. См. руководство cmake-generators(7) для документации по всем генераторам. Запустите cmake --help, чтобы увидеть список доступных локально генераторов. Вы можете использовать опцию -G для указания генератора, или просто принять генератор по умолчанию, выбранный CMake для текущей платформы.

При использовании одного из генераторов командной строки CMake ожидает, что среда, необходимая для инструментов компиляции, уже настроена в оболочке. При использовании одного из генераторов IDE специальная настройка среды не требуется.

Генерация Системы Сборки Проекта

Запустите CMake с одной из следующих команд, чтобы указать директории исходников и сборки, а также сгенерировать систему сборки:

cmake [<options>] -B <path-to-build> [-S <path-to-source>]

Новое в версии 3.13.

Использует <path-to-build> как директорию сборки и <path-to-source> как директорию с исходными кодами. Указанные пути могут быть абсолютными или относительными от текущей рабочей директории. В директории с исходными кодами должен находиться файл CMakeLists.txt. Директория для сборки будет создана автоматически, если она еще не существует. Например:

$cmake -S src -B build
cmake [<options>] <path-to-source>

Использует текущую рабочую директорию как директорию сборки, а <path-to-source> как директорию с исходными кодами. Указанный путь может быть абсолютным или относительным от текущей рабочей директории. Директория с исходными кодами должна содержать файл CMakeLists.txt и не должна содержать файл CMakeCache.txt, так как последний указывает на уже существующую директорию сборки. Например:

$mkdir build ; cd build
$cmake ../src
cmake [<options>] <path-to-existing-build>

Использует <path-to-existing-build> как директорию сборки и загружает путь к исходникам из файла CMakeCache.txt, который должен был быть создан предыдущим запуском CMake. Указанный путь может быть абсолютным или относительным от текущей рабочей директории. Например: 

$cd build
$cmake .

Во всех случаях <options> могут включать ноль или более параметров из списка ниже.

Вышеописанные способы указания директорий исходников и сборки могут быть смешаны. Пути, указанные с -S или -B, всегда классифицируются как директории исходников или сборки соответственно. Пути, указанные без параметров, классифицируются на основе их содержимого и типа путей, указанных ранее. Если указан только один тип пути, текущая рабочая директория (cwd) используется для другого. Например:

Команда Директория исходников Директория сборки
cmake -B build cwd build
cmake -B build src src build
cmake -B build -S src src build
cmake src src cwd
cmake build (existing) loaded build
cmake -S src src cwd
cmake -S src build src build
cmake -S src -B build src build

Изменено в версии 3.23: CMake предупреждает, если указано несколько путей к исходникам. Это никогда официально не документировалось и не поддерживалось, но в старых версиях случайно принимались несколько путей и использовался последний указанный путь. Избегайте передачи нескольких путей к исходникам.

После генерации системы сборки можно использовать соответствующий инструмент сборки для сборки проекта. Например, после использования генератора Unix Makefiles можно напрямую выполнить команду make: 

make
make install

Кроме того, можно использовать CMake для сборки проекта, автоматически выбирая и вызывая соответствующий инструмент сборки.

Опции

-S <path-to-source>

Путь к корневой директории проекта CMake, который нужно собрать.

-B <path-to-build>

Путь к директории, которую CMake будет использовать как корневую директорию сборки.

Если директория еще не существует, CMake создаст ее автоматически.

-C <initial-cache>

Предварительно загрузить скрипт для заполнения кэша.

Когда CMake запускается впервые в пустой директории сборки, он создает файл CMakeCache.txt и заполняет его настройками проекта, которые могут быть изменены. Эта опция позволяет указать файл, из которого будут загружены записи кэша перед первым проходом по файлам CMake проекта. Загруженные записи имеют приоритет над значениями по умолчанию, заданными в проекте. Указанный файл должен быть скриптом CMake, содержащим команды set() с использованием опции CACHE, а не файлом формата кэша.

Ссылки на CMAKE_SOURCE_DIR и CMAKE_BINARY_DIR в скрипте будут оцениваться как верхнеуровневые директории исходников и сборки соответственно.

-D <var>:<type>=<value>, -D <var>=<value>

Создать или обновить запись в кэше CMake.

Когда CMake запускается впервые в пустой директории сборки, он создает файл CMakeCache.txt и заполняет его настраиваемыми параметрами проекта. Эта опция может быть использована для задания параметра, который будет иметь приоритет над значением по умолчанию в проекте. Опцию можно повторять столько раз, сколько нужно для добавления записей в кэш.

Если указана часть :<type>, она должна соответствовать одному из типов, указанных в документации по команде set() для её подписи CACHE. Если часть :<type> опущена, запись будет создана без типа, если она еще не существует с каким-либо типом. Если команда в проекте устанавливает тип как PATH или FILEPATH, то <value> будет преобразовано в абсолютный путь.

Эту опцию также можно указать как один аргумент: -D<var>:<type>=<value> или -D<var>=<value>.

Важно отметить, что порядок аргументов -C и -D имеет значение. Они будут выполнены в том порядке, в котором они указаны, и последний аргумент имеет приоритет над предыдущими. Например, если вы укажете -DCMAKE_BUILD_TYPE=Debug, а затем аргумент -C с файлом, который вызывает:

set(CMAKE_BUILD_TYPE "Release" CACHE STRING "" FORCE)

то аргумент -C будет иметь приоритет, и CMAKE_BUILD_TYPE будет установлено в Release. Однако, если аргумент -D указан после аргумента -C, он будет установлен в Debug.

Если вызов set(... CACHE ...) в файле -C не использует FORCE, и аргумент -D устанавливает ту же переменную, аргумент -D будет иметь приоритет независимо от порядка из-за природы вызовов set(... CACHE ...) без FORCE.

-U <globbing_expr>

Удалить соответствующие записи из кэша CMake.

Эта опция может использоваться для удаления одной или нескольких переменных из файла CMakeCache.txt. Поддерживаются выражения с подстановочными знаками * и ?. Опция может быть повторена столько раз, сколько нужно для удаления записей из кэша.

Используйте осторожно, вы можете сделать ваш CMakeCache.txt нерабочим.

-G <generator-name>

Указать генератор системы сборки.

CMake может поддерживать несколько нативных систем сборки на определенных платформах. Генератор отвечает за создание конкретной системы сборки. Возможные имена генераторов указаны в руководстве cmake-generators(7).

Если не указано, CMake проверяет переменную окружения CMAKE_GENERATOR и в противном случае использует встроенный выбор по умолчанию.

-T <toolset-spec>

Спецификация набора инструментов для генератора, если поддерживается.

Некоторые генераторы CMake поддерживают спецификацию набора инструментов для указания нативной системе сборки, какой компилятор использовать. См. переменную CMAKE_GENERATOR_TOOLSET для получения подробной информации.

-A <platform-name>

Указать имя платформы, если это поддерживается генератором.

Некоторые генераторы CMake поддерживают указание имени платформы, которое будет передано нативной системе сборки для выбора компилятора или SDK. См. переменную CMAKE_GENERATOR_PLATFORM для получения подробной информации.

--toolchain <path-to-file>

Новое в версии 3.21.

Указать файл с цепочкой инструментов для кросс-компиляции, что эквивалентно установке переменной CMAKE_TOOLCHAIN_FILE. Относительные пути интерпретируются как относительные от директории сборки, а если файл не найден, то от директории исходников.

--install-prefix <directory>

Новое в версии 3.21.

Указать директорию установки, которая будет использоваться переменной CMAKE_INSTALL_PREFIX. Должен быть абсолютный путь.

-Wno-dev

Подавить предупреждения для разработчиков.

Подавить предупреждения, предназначенные для автора файлов CMakeLists.txt. По умолчанию это также отключает предупреждения об устаревании.

-Wdev

Включить предупреждения для разработчиков.

Включить предупреждения, предназначенные для автора файлов CMakeLists.txt. По умолчанию это также включает предупреждения об устаревании.

-Wdeprecated

Включить предупреждения об устаревших функциях.

Включить предупреждения об использовании устаревших функций, предназначенные для автора файлов CMakeLists.txt.

-Wno-deprecated

Подавить предупреждения об устаревших функциях.

Подавить предупреждения об использовании устаревших функций, предназначенные для автора файлов CMakeLists.txt.

-Werror=<what>

Обрабатывать предупреждения CMake как ошибки. <what> должен быть одним из следующих:

  • dev: Превратить предупреждения для разработчиков в ошибки.

    • Сделать ошибки из предупреждений, предназначенных для автора файлов CMakeLists.txt. По умолчанию это также включает превращение предупреждений об устаревших функциях в ошибки.

  • deprecated: Превратить предупреждения об устаревших макросах и функциях в ошибки.

    • Сделать ошибки из предупреждений об использовании устаревших макросов и функций, предназначенных для автора файлов CMakeLists.txt.

-Wno-error=<what>

Не обрабатывать предупреждения CMake как ошибки. <what> должен быть одним из следующих:

  • dev: Сделать предупреждения, предназначенные для автора файлов CMakeLists.txt, не ошибками. По умолчанию это также отключит предупреждения об устаревших функциях как ошибки.
  • deprecated: Сделать предупреждения об использовании устаревших макросов и функций, предназначенные для автора файлов CMakeLists.txt, не ошибками.
--fresh

Новое в версии 3.24.

Выполнить новую конфигурацию дерева сборки. Это удалит любой существующий файл CMakeCache.txt и связанную директорию CMakeFiles/, и пересоздаст их с нуля.

Изменено в версии 3.30: Для зависимостей, ранее загруженных с помощью FetchContent с новым параметром для политики CMP0168, их файлы штампов и сценариев из любого предыдущего запуска будут удалены. Шаги загрузки, обновления и патча будут вынуждены повторно выполниться.

-L[A][H]

Список нерасширенных кэшированных переменных.

Список кэшированных переменных запустит CMake и отобразит все переменные из кэша CMake, которые не помечены как INTERNAL или ADVANCED. Это эффективно покажет текущие настройки CMake, которые затем могут быть изменены с помощью опции -D. Изменение некоторых переменных может привести к созданию дополнительных переменных. Если указано A, то также будут отображены расширенные переменные. Если указано H, будет показана помощь для каждой переменной.

-N

Только режим просмотра.

Загрузить только кэш. Фактические шаги настройки и генерации не будут выполнены.

--graphviz=<file>

Создать график зависимостей в формате graphviz, см. CMakeGraphVizOptions для получения дополнительной информации.

Создать входной файл для graphviz, который будет содержать все зависимости библиотек и исполняемых файлов в проекте. См. документацию для CMakeGraphVizOptions для получения дополнительных сведений.

--system-information [file]

Вывести информацию об этой системе.

Вывести широкий диапазон информации о текущей системе. Если запустить из верхней директории бинарного дерева для проекта CMake, будет выведена дополнительная информация, такая как кэш, файлы логов и т.д.

--log-level=<level>

Установить уровень журнала <level>.

Команда message() будет выводить только сообщения указанного уровня журнала или выше. Допустимые уровни журнала: ERROR, WARNING, NOTICE, STATUS (по умолчанию), VERBOSE, DEBUG или TRACE.

Чтобы сделать уровень журнала постоянным между запусками CMake, установите CMAKE_MESSAGE_LOG_LEVEL в качестве переменной кэша. Если указаны как опция командной строки, так и переменная, то опция командной строки имеет приоритет.

Новое в версии 3.25: См. команду cmake_language() для запроса текущего уровня ведения журнала сообщений.

--log-context

Включить вывод команды message() с контекстом, прикрепленным к каждому сообщению.

Эта опция включает отображение контекста только для текущего запуска CMake. Чтобы сделать отображение контекста постоянным для всех последующих запусков CMake, установите CMAKE_MESSAGE_CONTEXT_SHOW в качестве переменной кэша. При наличии этой опции командной строки, CMAKE_MESSAGE_CONTEXT_SHOW игнорируется.

--debug-trycompile

Не удалять файлы и директории, созданные для вызовов try_compile() / try_run(). Это полезно для отладки неудачных проверок.

Учтите, что некоторые вызовы try_compile() могут использовать одно и то же дерево сборки, что ограничивает полезность этой опции, если проект выполняет более одного try_compile(). Например, такие вызовы могут изменить результаты, так как артефакты из предыдущего компилятора могут привести к некорректному завершению другого теста. Эта опция лучше всего подходит только для отладки.

Новое в версии 3.25: Когда эта опция включена, каждая проверка компиляции выводит сообщение журнала, сообщающее о директории, в которой выполняется проверка.

--debug-output

Запустить cmake в режиме отладки.

Выводить дополнительную информацию во время выполнения cmake, например, трассировку стека при вызовах message(SEND_ERROR).

--debug-find

Новое в версии 3.17.

Запустить cmake find команды в режиме отладки.

Выводить дополнительную информацию о вызовах find во время выполнения cmake в стандартный поток ошибок. Вывод предназначен для восприятия человеком и не для парсинга. См. также переменную CMAKE_FIND_DEBUG_MODE для отладки локальной части проекта.

--debug-find-pkg=<pkg>[,...]

Новое в версии 3.23.

Запустить cmake find команды в режиме отладки при вызовах find_package(<pkg>), где <pkg> — это запись в указанном через запятую списке регистрозависимых имен пакетов.

Как --debug-find, но с ограничением области действия до указанных пакетов.

--debug-find-var=<var>[,...]

Новое в версии 3.23.

Запустить cmake find команды в режиме отладки при вызовах с <var> в качестве результирующей переменной, где <var> — это запись в указанном через запятую списке имен переменных.

Как --debug-find, но с ограничением области действия до указанных имен переменных.

--trace

Запустить cmake в режиме трассировки.

Выводить трассировку всех вызовов и их источников.

--trace-expand

Запустить cmake в режиме трассировки.

Как --trace, но с разворачиванием переменных

--trace-format=<format>

Новое в версии 3.17.

Запустить cmake в режиме трассировки и установить формат вывода трассировки.

<format> может быть одним из следующих значений:

  • human: Выводит каждую строку трассировки в формате, читаемом человеком. Это формат по умолчанию.
  • json-v1: Выводит каждую строку в виде отдельного документа JSON. Каждый документ разделяется новой строкой (\n). Гарантируется, что внутри документа JSON не будет символов новой строки.

Формат JSON трассировки:

{
  "file": "/full/path/to/the/CMake/file.txt",
  "line": 0,
  "cmd": "add_executable",
  "args": ["foo", "bar"],
  "time": 1579512535.9687231,
  "frame": 2,
  "global_frame": 4
}

Поля:

  • file: Полный путь к исходному файлу CMake, в котором была вызвана функция.
  • line: Строка в файле, с которой начинается вызов функции.
  • line_end: Если вызов функции занимает несколько строк, это поле будет содержать строку, на которой заканчивается вызов функции. Если вызов функции занимает одну строку, это поле не будет установлено. Это поле добавлено в малой версии 2 формата json-v1.
  • defer: Дополнительное поле, присутствующее, когда вызов функции был отложен с помощью cmake_language(DEFER). Если присутствует, его значение — строка, содержащая идентификатор отложенного вызова.
  • cmd: Имя вызванной функции.
  • args: Строковый список всех параметров функции.
  • time: Временная метка (в секундах с начала эпохи) вызова функции.
  • frame: Глубина стека вызова функции в контексте текущего обрабатываемого CMakeLists.txt.
  • global_frame: Глобальная глубина стека вызова функции, отслеживаемая по всем файлам CMakeLists.txt, участвующим в трассировке. Это поле добавлено в малой версии 2 формата json-v1.

Кроме того, первый выводимый документ JSON содержит ключ version для текущей основной и малой версии формата JSON:

{
  "version": {
   "major": 1,
   "minor": 2
  }
}

Поля:

  • version: Указывает версию формата JSON. Версия имеет основные и малые компоненты, следующие семантическим соглашениям о версиях.
--trace-source=<file>

Запустить cmake в режиме трассировки, но выводить только строки указанного файла.

Разрешены несколько опций.

--trace-redirect=<file>

Запустить cmake в режиме трассировки и перенаправить вывод трассировки в файл вместо стандартного потока ошибок.

--warn-uninitialized

Предупреждать об использовании неинициализированных значений.

Выводить предупреждение, когда используется неинициализированная переменная.

--warn-unused-vars

Не выполняет никаких действий. В версиях CMake 3.2 и ниже это включало предупреждения о неиспользуемых переменных. В версиях CMake 3.3–3.18 опция была неисправной. В версиях CMake 3.19 и выше опция была удалена.

--no-warn-unused-cli

Не предупреждать о параметрах командной строки.

Не искать переменные, объявленные в командной строке, но не использованные.

--check-system-vars

Найти проблемы с использованием переменных в системных файлах.

Обычно неиспользуемые и неинициализированные переменные ищутся только в CMAKE_SOURCE_DIR и CMAKE_BINARY_DIR. Этот флаг указывает CMake предупреждать и о других файлах.

--compile-no-warning-as-error

Новое в версии 3.24.

Игнорировать свойство цели COMPILE_WARNING_AS_ERROR и переменную CMAKE_COMPILE_WARNING_AS_ERROR, предотвращая обработку предупреждений как ошибок при компиляции.

--profiling-output=<path>

Новое в версии 3.18.

Используется вместе с --profiling-format для вывода данных профилирования в заданный путь.

--profiling-format=<file>

Включить вывод данных профилирования скрипта CMake в заданном формате.

Это может помочь в анализе производительности выполняемых скриптов CMake. Для обработки вывода в читаемый формат следует использовать сторонние приложения.

В настоящее время поддерживаются следующие значения: google-trace — вывод в формате Google Trace, который может быть проанализирован вкладкой about:tracing браузера Google Chrome или с помощью плагина для инструмента, такого как Trace Compass.

--preset <preset>, --preset=<preset>

Считывает предустановку из файлов CMakePresets.json и CMakeUserPresets.json, которые должны находиться в той же директории, что и файл CMakeLists.txt верхнего уровня. Предустановка может указывать генератор, директорию сборки, список переменных и другие аргументы для передачи CMake. По крайней мере один из файлов CMakePresets.json или CMakeUserPresets.json должен присутствовать. GUI CMake также распознает и поддерживает файлы CMakePresets.json и CMakeUserPresets.json. Для получения полной информации о этих файлах см. cmake-presets(7).

Предустановки читаются перед всеми другими параметрами командной строки, хотя опция -S может использоваться для указания директории исходного кода, содержащей файлы CMakePresets.json и CMakeUserPresets.json. Если -S не указана, текущая директория считается корневой директорией исходного кода и должна содержать файлы предустановок. Опции, указанные выбранной предустановкой (переменные, генератор и т.д.), могут быть переопределены вручную, указав их в командной строке. Например, если предустановка задает переменную MYVAR равной 1, но пользователь задает ее равной 2 с помощью аргумента -D, предпочтение будет отдано значению 2.

--list-presets[=<type>]

Список доступных предустановок указанного типа <type>. Допустимые значения для <type> — configure, build, test, package или all. Если <type> не указан, по умолчанию предполагается configure. Текущая рабочая директория должна содержать файлы предустановок CMake, если не используется опция -S для указания другой корневой директории исходного кода.

--debugger

Включает интерактивную отладку языка CMake. CMake предоставляет интерфейс отладки через канал, указанный в --debugger-pipe, который соответствует спецификации Debug Adapter Protocol с следующими модификациями.

Ответ на запрос инициализации содержит дополнительное поле cmakeVersion, которое указывает версию CMake, находящуюся под отладкой.

Ответ инициализации отладчика:

{
  "cmakeVersion": {
   "major": 3,
   "minor": 27,
   "patch": 0,
   "full": "3.27.0"
  }
}

Поля:

  • major: Целое число, указывающее основной номер версии.
  • minor: Целое число, указывающее номер минорной версии.
  • patch: Целое число, указывающее номер патча.
  • full: Строка, указывающая полную версию CMake.
--debugger-pipe <pipe name>, --debugger-pipe=<pipe name>

Имя канала (в Windows) или доменного сокета (в Unix), который используется для связи отладчика.

--debugger-dap-log <log path>, --debugger-dap-log=<log path>

Записывает все сообщения отладчика в указанный файл.

Сборка проекта

CMake предоставляет команду для сборки уже сгенерированного бинарного дерева проекта:

cmake --build <dir> [<options>] [-- <build-tool-options>]
cmake --build --preset <preset> [<options>] [-- <build-tool-options>]

Эта команда абстрагирует интерфейс командной строки нативного инструмента сборки с использованием следующих опций:

--build <dir>

Директория бинарного дерева проекта, которую необходимо собрать. Это обязательный параметр (если не указан preset) и должен быть первым.

--preset <preset>, --preset=<preset>

Использовать пресет сборки для указания параметров сборки. Директория бинарного дерева проекта определяется из ключа configurePreset. Текущая рабочая директория должна содержать файлы пресета CMake. См. preset для подробностей.

--list-presets

Показать доступные пресеты сборки. Текущая рабочая директория должна содержать файлы пресета CMake.

-j [<jobs>], --parallel [<jobs>]

Новое в версии 3.12.

Максимальное количество параллельных процессов для использования при сборке. Если <jobs> не указано, используется значение по умолчанию нативного инструмента сборки.

Переменная среды CMAKE_BUILD_PARALLEL_LEVEL, если установлена, задает значение по умолчанию для параллелизма, если эта опция не указана.

Некоторые нативные инструменты сборки всегда работают в параллельном режиме. Использование значения <jobs> равного 1 может ограничить выполнение до одного процесса.

-t <tgt>..., --target <tgt>...

Собрать <tgt> вместо целевого по умолчанию. Можно указать несколько целей, разделенных пробелами.

--config <cfg>

Для много-конфигурационных инструментов выберите конфигурацию <cfg>.

--clean-first

Сначала собрать цель clean, затем собрать. (Для очистки без сборки используйте --target clean.)

--resolve-package-references=<value>

Новое в версии 3.23.

Разрешить удаленные ссылки на пакеты из внешних менеджеров пакетов (например, NuGet) перед сборкой. Если значение <value> установлено в on (по умолчанию), пакеты будут восстановлены перед сборкой цели. Если значение установлено в only, пакеты будут восстановлены, но сборка не будет выполняться. Если значение установлено в off, пакеты не будут восстанавливаться.

Если цель не определяет никаких ссылок на пакеты, эта опция ничего не делает.

Эта настройка может быть указана в пресете сборки (с использованием resolvePackageReferences). Настройка пресета будет игнорироваться, если указана эта опция командной строки.

Если не указаны параметры командной строки или параметры пресета, будет оцениваться специфическая для среды переменная кеша, чтобы решить, следует ли выполнять восстановление пакетов.

При использовании генератора Visual Studio ссылки на пакеты определяются с использованием свойства VS_PACKAGE_REFERENCES. Восстановление пакетов выполняется с использованием NuGet. Оно может быть отключено путем установки переменной CMAKE_VS_NUGET_PACKAGE_RESTORE в значение OFF.

--use-stderr

Игнорируется. Поведение по умолчанию в CMake >= 3.0.

-v, --verbose

Включить подробный вывод - если поддерживается - включая команды сборки, которые будут выполнены.

Эта опция может быть опущена, если установлена переменная среды VERBOSE или кеш-переменная CMAKE_VERBOSE_MAKEFILE.

--

Передать оставшиеся опции нативному инструменту.

Запустите cmake --build без параметров для быстрого получения справки.

Установка проекта

CMake предоставляет команду для установки уже сгенерированного бинарного дерева проекта:

cmake --install <dir> [<options>]

Эта команда может использоваться после сборки проекта для выполнения установки без использования сгенерированной системы сборки или нативного инструмента сборки. Опции:

--install <dir>

Директория бинарного дерева проекта для установки. Это обязательный параметр и должен быть первым.

--config <cfg>

Для много-конфигурационных генераторов выберите конфигурацию <cfg>.

--component <comp>

Установка на основе компонентов. Установить только компонент <comp>.

--default-directory-permissions <permissions>

Права доступа по умолчанию для установки директорий. Формат прав: <u=rwx,g=rx,o=rx>.

--prefix <prefix>

Переопределить префикс установки, CMAKE_INSTALL_PREFIX.

--strip

Выполнить strip перед установкой.

-v, --verbose

Включить подробный вывод.

Эта опция может быть опущена, если установлена переменная среды VERBOSE.

Запустите cmake --install без параметров для быстрого получения справки.

Открытие проекта

cmake --open <dir>

Открыть сгенерированный проект в ассоциированном приложении. Это поддерживается только некоторыми генераторами.

Запуск скрипта

cmake [-D <var>=<value>]... -P <cmake-script-file> [-- <unparsed-options>...]
-D <var>=<value>

Определить переменную для режима выполнения скрипта.

-P <cmake-script-file>

Обработать указанный файл CMake как скрипт, написанный на языке CMake. Шаги конфигурации и генерации не выполняются, и кеш не изменяется. Если переменные определены с помощью -D, это должно быть сделано перед аргументом -P.

Любые опции после -- не анализируются CMake, но они все равно включаются в набор переменных CMAKE_ARGV<n>, передаваемых скрипту (включая сам --).

Запуск инструментов командной строки

CMake предоставляет встроенные командные утилиты через следующую сигнатуру:

cmake -E <command> [<options>]
-E [help]

Запуск cmake -E или cmake -E help для получения краткого описания команд.

Доступные команды:

capabilities

Новая команда в версии 3.7.

Выводит возможности CMake в формате JSON. Результат представляет собой JSON-объект со следующими ключами:

  • version: JSON-объект с информацией о версии. Ключи:
    • string: Полная строка версии, как отображается при выполнении cmake --version.
    • major: Основной номер версии в виде целого числа.
    • minor: Второстепенный номер версии в виде целого числа.
    • patch: Патч-уровень в виде целого числа.
    • suffix: Суффикс версии CMake.
    • isDirty: Логическое значение, указывающее на сборку из "грязного" дерева исходного кода.
  • generators: Список доступных генераторов. Каждый генератор — это JSON-объект с ключами:
    • name: Имя генератора.
    • toolsetSupport: Логическое значение, указывающее поддержку наборов инструментов.
    • platformSupport: Логическое значение, указывающее поддержку платформ.
    • supportedPlatforms: Новая опция в версии 3.21. Список поддерживаемых платформ.
    • extraGenerators: Список дополнительных генераторов, совместимых с основным генератором.
  • fileApi: Опциональный член, присутствующий при наличии cmake-file-api(7). Значение — JSON-объект с ключом requests, содержащим массив поддерживаемых запросов.
  • serverMode: Логическое значение, указывающее на поддержку серверного режима. Всегда false с версии CMake 3.20.
  • tls: Новая опция в версии 3.25. Логическое значение, указывающее наличие поддержки TLS.
  • debugger: Новая опция в версии 3.27. Логическое значение, указывающее поддержку режима отладки.

cat [--] <files>...

Новая команда в версии 3.18.

Конкатенирует файлы и выводит их содержимое в стандартный вывод.

  • --: Новая опция в версии 3.24. Указывает конец опций, что позволяет использовать файлы, начинающиеся с символа -.
  • Новая опция в версии 3.29: возможность вывода стандартного ввода при передаче аргумента -.

chdir <dir> <cmd> [<arg>...]

Изменяет текущий рабочий каталог и выполняет команду.

compare_files [--ignore-eol] <file1> <file2>

Проверяет, являются ли <file1> и <file2> одинаковыми. Возвращает 0, если они одинаковые, 1 — если нет, 2 — в случае неверных аргументов.

  • --ignore-eol: Новая опция в версии 3.14. Игнорирует различия в окончаниях строк (LF/CRLF) при построчном сравнении.

copy <file>... <destination>, copy -t <destination> <file>...

Копирует файлы в <destination> (файл или каталог). Если указано несколько файлов или аргумент -t, <destination> должен быть существующим каталогом.

  • Новая опция в версии 3.5: Поддержка нескольких входных файлов.
  • Новая опция в версии 3.26: Поддержка аргумента -t.

copy_directory <dir>... <destination>

Копирует содержимое каталогов <dir>... в каталог <destination>. Если каталог <destination> не существует, он будет создан. Команда copy_directory следует за символическими ссылками.

  • Новая опция в версии 3.5: Поддержка нескольких входных каталогов.
  • Новая опция в версии 3.15: Команда теперь завершается с ошибкой, если исходный каталог не существует. Ранее она успешно завершалась, создавая пустой каталог назначения.

copy_directory_if_different <dir>... <destination>

Новая команда в версии 3.26.

Копирует измененное содержимое каталогов <dir>... в каталог <destination>. Если каталог <destination> не существует, он будет создан. Команда copy_directory_if_different следует за символическими ссылками. Команда завершается с ошибкой, если исходный каталог не существует.

copy_if_different <file>... <destination>

Копирует файлы в <destination> (файл или каталог) только в случае, если они изменились. Если указано несколько файлов, <destination> должен быть существующим каталогом. Команда copy_if_different следует за символическими ссылками.

  • Новая опция в версии 3.5: Поддержка нескольких входных файлов.

create_symlink <old> <new>

Создает символическую ссылку <new>, указывающую на <old>.

  • Новая опция в версии 3.13: Поддержка создания символических ссылок в Windows.
Примечание: Путь к месту, где будет создана символическая ссылка <new>, должен существовать заранее.

create_hardlink <old> <new>

Новая команда в версии 3.19.

Создает жесткую ссылку <new>, указывающую на <old>.

Примечание: Путь к месту, где будет создана жесткая ссылка <new>, должен существовать заранее. <old> должен существовать заранее.

echo [<string>...]

Выводит аргументы в виде текста.

echo_append [<string>...]

Выводит аргументы в виде текста без добавления новой строки.

env [<options>] [--] <command> [<arg>...]

Новая команда в версии 3.1.

Запускает команду в измененной среде. Доступные опции:

  • NAME=VALUE: Заменяет текущее значение переменной среды NAME на VALUE.
  • --unset=NAME: Удаляет текущее значение переменной среды NAME.
  • --modify ENVIRONMENT_MODIFICATION: Новая опция в версии 3.25. Применяет единичную операцию модификации среды к измененной среде. Опции NAME=VALUE и --unset=NAME эквивалентны --modify NAME=set:VALUE и --modify NAME=unset: соответственно. Обратите внимание, что --modify NAME=reset: сбрасывает NAME к значению, которое оно имело при запуске CMake (или удаляет его), а не к последнему значению, установленному через NAME=VALUE.
  • --: Новая опция в версии 3.24. Добавлена поддержка аргумента с двойным дефисом --. Используйте --, чтобы прекратить интерпретацию опций/переменных среды и рассматривать следующий аргумент как команду, даже если он начинается с - или содержит =.

remove [-f] <file>...

Устаревшая команда с версии 3.17.

Удаляет файл(ы). Планировалось, что если любой из перечисленных файлов не существует, команда вернет код выхода, не равный нулю, но без вывода сообщений. Опция -f изменяет поведение, возвращая нулевой код выхода (т.е. успех) в таких ситуациях. Команда remove не следует за символическими ссылками, то есть она удаляет только символические ссылки, а не файлы, на которые они указывают.

Важно: Реализация была ошибочной и всегда возвращала 0. Это не может быть исправлено без нарушения обратной совместимости. Используйте rm вместо этого.

remove_directory <dir>...

Устаревшая команда с версии 3.17.

Удаляет каталоги <dir>... и их содержимое. Если каталог не существует, он будет проигнорирован без сообщений. Используйте rm вместо этого.

  • Новая опция в версии 3.15: Поддержка нескольких каталогов.
  • Новая опция в версии 3.16: Если <dir> является символической ссылкой на каталог, будет удалена только символическая ссылка.

rename <oldname> <newname>

Переименовывает файл или каталог (на одном томе). Если файл с именем <newname> уже существует, он будет тихо заменен.

rm [-rRf] [--] <file|dir>...

Новая команда в версии 3.17.

Удаляет файлы <file> или каталоги <dir>. Используйте -r или -R для рекурсивного удаления каталогов и их содержимого. Если любой из перечисленных файлов/каталогов не существует, команда возвращает код выхода, не равный нулю, но без вывода сообщений. Опция -f изменяет поведение, возвращая нулевой код выхода (т.е. успех) в таких ситуациях. Используйте --, чтобы прекратить интерпретацию опций и рассматривать все оставшиеся аргументы как пути, даже если они начинаются с -.

sleep <number>

Новая команда в версии 3.0.

Приостанавливает выполнение на <number> секунд. <number> может быть числом с плавающей точкой. Практический минимум составляет около 0.1 секунды из-за накладных расходов на запуск/остановку исполняемого файла CMake. Это может быть полезно для вставки задержки в скрипте CMake:

# Задержка примерно на 0.5 секунд
execute_process(COMMAND ${CMAKE_COMMAND} -E sleep 0.5)

tar [cxt][vf][zjJ] file.tar [<options>] [--] [<pathname>...]

Создает или извлекает архив tar или zip. Доступные опции:

  • c: Создает новый архив, содержащий указанные файлы. При использовании аргумент <pathname>... обязателен.
  • x: Извлекает содержимое архива на диск.
    • Новая опция в версии 3.15: Аргумент <pathname>... может использоваться для извлечения только выбранных файлов или каталогов. При извлечении выбранных файлов или каталогов вы должны предоставить их точные имена, включая путь, как отображается при перечислении содержимого (флаг -t).
  • t: Перечисляет содержимое архива.
    • Новая опция в версии 3.15: Аргумент <pathname>... может использоваться для перечисления только выбранных файлов или каталогов.
  • v: Выводит подробную информацию.
  • z: Сжимает результирующий архив с помощью gzip.
  • j: Сжимает результирующий архив с помощью bzip2.
  • J: Новая опция в версии 3.1. Сжимает результирующий архив с помощью XZ.
  • --zstd: Новая опция в версии 3.15. Сжимает результирующий архив с помощью Zstandard.
  • --files-from=<file>: Новая опция в версии 3.1. Читает имена файлов из указанного файла, по одному на строку. Пустые строки игнорируются. Строки не могут начинаться с -, за исключением --add-file=<name> для добавления файлов, имена которых начинаются с -.
  • --format=<format>: Новая опция в версии 3.3. Указывает формат создаваемого архива. Поддерживаемые форматы: 7zip, gnutar, pax, paxr (ограниченный pax, по умолчанию) и zip.
  • --mtime=<date>: Новая опция в версии 3.1. Указывает время модификации, записываемое в записи tarball.
  • --touch: Новая опция в версии 3.24. Использует текущую локальную временную метку вместо извлечения временных меток файлов из архива.
  • --: Новая опция в версии 3.1. Прекращает интерпретацию опций и рассматривает все оставшиеся аргументы как имена файлов, даже если они начинаются с -.
  • Новая опция в версии 3.1: Поддержка LZMA (7zip).
  • Новая опция в версии 3.15: Команда теперь продолжает добавлять файлы в архив, даже если некоторые из файлов недоступны для чтения. Такое поведение более согласуется с классическим инструментом tar. Команда также теперь анализирует все флаги, и если предоставлен недопустимый флаг, выдается предупреждение.

time <command> [<args>...]

Выполняет <command> и отображает затраченное время (включая накладные расходы интерфейса CMake).

  • Новая опция в версии 3.5: Команда теперь правильно передает аргументы с пробелами или специальными символами в дочерний процесс. Это может нарушить работу скриптов, которые обходили ошибку с помощью дополнительного экранирования или кавычек.

touch <file>...

Создает файлы <file>..., если они не существуют. Если файл существует, изменяет время доступа и модификации файла.

touch_nocreate <file>...

Изменяет временные метки файла, если он существует, но не создает его. Если файл не существует, он будет проигнорирован без сообщений.

true

Ничего не делает, возвращая код выхода 0.

Запуск инструмента Find-Package

CMake предоставляет аналог инструмента pkg-config для проектов на основе Makefile:

cmake --find-package [<опции>]

Ищет пакет с помощью find_package() и выводит результат в stdout. Это можно использовать вместо pkg-config для поиска установленных библиотек в проектах на основе Makefile или в проектах на основе autoconf (через share/aclocal/cmake.m4).

Примечание: Этот режим плохо поддерживается из-за некоторых технических ограничений. Он сохраняется для совместимости, но не должен использоваться в новых проектах.

Запуск предустановок рабочего процесса

Новое в версии 3.25.

Предустановки CMake предоставляют способ выполнения нескольких этапов сборки по порядку:

cmake --workflow [<опции>]

Опции:

  • --workflow: Выбрать предустановку рабочего процесса, используя одну из следующих опций.
  • --preset <предустановка>, --preset=<предустановка>: Использовать предустановку рабочего процесса для указания рабочего процесса. Директория бинарного проекта определяется исходя из начальной конфигурации предустановки. Текущая рабочая директория должна содержать файлы предустановок CMake. См. preset для получения более подробной информации.
  • --list-presets: Список доступных предустановок рабочего процесса. Текущая рабочая директория должна содержать файлы предустановок CMake.
  • --fresh: Провести новую конфигурацию дерева сборки, что имеет тот же эффект, что и cmake --fresh.

Просмотр справки

Для печати выбранных страниц из документации CMake используйте команду

cmake --help[-<тема>]

с одной из следующих опций:

  • -version [<файл>], --version [<файл>], /V [<файл>]: Показать баннер с названием программы/версией и выйти. Вывод записывается в указанный <файл>, если он указан.
  • -h, -H, --help, -help, -usage, /?: Показать справочную информацию и выйти.
  • --help <ключевое_слово> [<файл>]: Показать справку по одному ключевому слову CMake. <ключевое_слово> может быть свойством, переменной, командой, политикой, генератором или модулем. Соответствующая справочная запись выводится в читаемом формате. Вывод записывается в указанный <файл>, если он указан.
  • --help-full [<файл>]: Печать всех справочных руководств и выход.
  • --help-manual <руководство> [<файл>]: Печать одного справочного руководства и выход.
  • --help-manual-list [<файл>]: Список доступных справочных руководств и выход.
  • --help-command <команда> [<файл>]: Показать справку по одной команде и выйти.
  • --help-command-list [<файл>]: Список команд с доступной справкой и выход.
  • --help-commands [<файл>]: Печать руководства по командам CMake и выход.
  • --help-module <модуль> [<файл>]: Показать справку по одному модулю и выйти.
  • --help-module-list [<файл>]: Список модулей с доступной справкой и выход.
  • --help-modules [<файл>]: Печать руководства по модулям CMake и выход.
  • --help-policy <политика> [<файл>: Показать справку по одной политике и выйти.
  • --help-policy-list [<файл>]: Список политик с доступной справкой и выход.
  • --help-policies [<файл>]: Печать руководства по политикам CMake и выход.
  • --help-property <свойство> [<файл>]: Показать справку по одному свойству и выйти.
  • --help-property-list [<файл>]: Список свойств с доступной справкой и выход.
  • --help-properties [<файл>]: Печать руководства по свойствам CMake и выход.
  • --help-variable <переменная> [<файл>]: Показать справку по одной переменной и выйти.
  • --help-variable-list [<файл>]: Список переменных с доступной справкой и выход.
  • --help-variables [<файл>]: Печать руководства по переменным CMake и выход.

Для просмотра доступных предустановок проекта используйте команду

cmake <source-dir> --list-presets

Возвращаемое значение (Код завершения)

При нормальном завершении выполнение cmake возвращает код завершения 0.

Если завершение вызвано командой message(FATAL_ERROR) или другой ошибочной ситуацией, то возвращается ненулевой код завершения.
